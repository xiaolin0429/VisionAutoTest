# 前端联调总览

## 1. 适用范围

本文档面向当前 `VisionAutoTest` 前后端联调阶段，重点覆盖模板资产工作台相关的真实后端基线：

- 认证与工作空间隔离：`JWT + X-Workspace-Id`
- 媒体访问模型：统一走 `media-objects/{id}/content`
- 模板资产域：模板、基准版本、Mask 区域、OCR 快照、Mask 预览图
- 执行链路：`FastAPI + BackgroundTasks + Playwright`

本文档以当前后端代码实现、`doc/api/` 契约文档和本地联调环境为准。

## 2. 当前联调基线

- 接口前缀：`/api/v1`
- 健康检查：`GET /healthz`
- 默认管理员账号：`admin`
- 默认管理员密码：以本地 `backend/.env` 中 `VAT_DEFAULT_ADMIN_PASSWORD` 为准
- 工作空间内接口统一要求：
  - `Authorization: Bearer <token>`
  - `X-Workspace-Id`

当前模板资产工作台后端已支持：

- 基于模板当前基准图访问真实图片
- 基于显式 `baseline_revision_id` 发起 OCR 分析
- OCR 结果按 `template + baseline_revision` 唯一持久化
- OCR 失败态快照可回读，不再表现为“空白结果”
- OCR 返回像素坐标和比例坐标，前端可直接转 Mask 草稿
- 基于已保存 Mask 或前端草稿 Mask 生成预览图
- 预览结果统一复用 `media-objects` 稳定访问

## 3. 本轮后端变更对前端的影响

### 3.1 模板工作台上下文已收口到显式基准版本

模板工作台相关新接口全部围绕：

- `template_id`
- `baseline_revision_id`

前端切换基准版本后，后续 OCR 与预览请求都必须显式带上新的 `baseline_revision_id`，不能再只依赖模板详情里的“当前基准版本”做隐式推断。

### 3.2 OCR 结果已成为独立快照资源

当前后端已新增：

- `GET /templates/{template_id}/baseline-revisions/{baseline_revision_id}/ocr-results`
- `POST /templates/{template_id}/baseline-revisions/{baseline_revision_id}/ocr-results`

前端联调规则：

- 页面首次进入某个基准版本时，先尝试 `GET`
- 若返回 `200 + status=not_generated`，前端应展示“未生成 OCR”状态，而不是空列表成功态
- 用户手动触发 OCR 时调用 `POST`
- `POST` 成功后前端直接用响应体刷新结果面板，不需要再次补一次 `GET`
- 同一模板同一基准版本重复 OCR 时，后端会覆盖更新原快照，前端不需要处理历史版本列表

### 3.3 OCR 结果可以直接转 Mask 草稿

OCR 响应中每个 `blocks[]` 都包含：

- `polygon_points`
- `pixel_rect`
- `ratio_rect`

前端应优先使用 `ratio_rect` 创建 Mask 草稿，因为它与模板正式 `mask-regions` 使用的是同一套比例坐标体系。

### 3.4 Mask 预览已支持两种输入模式

当前后端已新增：

- `POST /templates/{template_id}/baseline-revisions/{baseline_revision_id}/preview-images`

预览接口支持两种模式：

1. 不传 `mask_regions`
   - 使用模板当前已保存的正式 Mask
2. 传 `mask_regions`
   - 使用前端当前草稿 Mask
   - 不写回数据库

这意味着前端可以在“未保存草稿”状态下直接预览处理效果，不需要先保存再预览。

### 3.5 预览结果不再要求前端做图像计算

预览响应固定返回：

- `overlay_media_object_id`
- `overlay_content_url`
- `processed_media_object_id`
- `processed_content_url`
- `image_width`
- `image_height`
- `mask_regions`

前端只需要展示返回图片，不需要自行叠加 Mask 或自行做图像黑化处理。

## 4. 本地环境约定

### 4.1 数据库

前后端本地联调默认使用 PostgreSQL：

- 管理库：`postgres`
- 开发库：`AutoTestDev`
- 默认账号：`postgres`
- 默认密码：`postgres`
- 端口：`5432`

本地业务库连接示例：

```text
postgresql+psycopg://postgres:postgres@127.0.0.1:5432/AutoTestDev
```

### 4.2 后端初始化

联调前请先完成：

```bash
cd backend
source .venv/bin/activate
python -m app.db.bootstrap
uvicorn app.main:app --reload
```

### 4.3 视觉依赖

当前联调若涉及 `template_assert / ocr_assert / 模板工作台 OCR / Mask 预览`，后端运行机必须具备：

- `opencv-python-headless`
- `paddleocr`
- `paddlepaddle`
- `playwright` 浏览器依赖

建议安装方式：

```bash
cd backend
source .venv/bin/activate
pip install -e ".[dev]"
playwright install chromium
```

## 5. 推荐联调顺序

### 5.1 模板工作台主链路

建议前端按以下顺序联调：

1. 登录
2. 创建工作空间
3. 上传模板原图
4. 创建模板
5. 读取模板详情，拿到 `current_baseline_revision_id`
6. 用模板原图内容接口展示真实图片
7. 对当前基准版本执行 OCR
8. 展示 OCR 结果列表和 OCR 框
9. 从 OCR 结果生成一个 Mask 草稿
10. 用草稿 Mask 调预览接口
11. 保存正式 Mask
12. 不传 `mask_regions` 再调一次预览接口，验证正式 Mask 预览

### 5.2 基准版本切换链路

建议前端补测：

1. 切换到另一个 `baseline_revision_id`
2. OCR 面板先重新走 `GET`
3. 若该版本未生成 OCR，展示未生成态
4. 预览接口改用新的 `baseline_revision_id`
5. 已保存 Mask 继续显示，但预览结果应基于新基准图

## 6. 前端页面联调关注点

### 6.1 模板图片显示

- 模板真实图片仍通过已有媒体访问模型获取
- 前端应先读模板详情，再根据 `current_baseline_revision_id` 读取基准版本列表或已有上下文，拿到 `media_object_id`
- 最终图片 URL 仍使用 `/api/v1/media-objects/{media_object_id}/content`

### 6.2 OCR 面板

前端需要明确区分以下三种状态：

- 未生成：`GET` 返回 `200 + status=not_generated`
- 成功：响应 `status=succeeded`
- 失败：响应 `status=failed`，同时带 `error_code`、`error_message`

### 6.3 Mask 编辑器

前端当前应继续以比例坐标作为正式持久化口径：

- `x_ratio`
- `y_ratio`
- `width_ratio`
- `height_ratio`

从 OCR 框生成 Mask 草稿时，直接使用 `ratio_rect` 即可。

### 6.4 预览视图

建议前端至少支持：

- 原图：基准图原始图片
- OCR 叠加：前端自行画框展示 OCR block
- Mask 叠加：使用 `overlay_content_url`
- Mask 处理后：使用 `processed_content_url`

`processed_content_url` 对应的图片由后端生成玻璃态 Mask 展示效果。前端仍按普通媒体对象图片展示，不需要额外重绘；联调时需确认 Mask 区域不是黑色实心块，底图内容仍可辨认。

### 6.5 执行报告 Mask 证据图

执行报告中的 `actual_screenshot` 是用户可见证据图。带 Mask 的模板断言应展示为“实际截图 + 玻璃态 Mask 覆盖”，而不是算法内部置黑后的中间图。

- Mask 区域内的变化仍不参与比对失败判定。
- Mask 区域外的变化仍按原规则生成失败和 Diff。
- `diff_screenshot` 可叠加玻璃态忽略区提示，但接口字段、媒体对象访问路径和 `artifact_type` 不变。
- 历史报告不会回刷，新生成报告产物使用新的玻璃态展示效果。

## 7. 联调排查重点

### 7.1 常见错误码

- `BASELINE_REVISION_NOT_FOUND`
  - 当前 `baseline_revision_id` 不存在，或不属于当前模板
- `TEMPLATE_OCR_RESULT_NOT_FOUND`
  - 当前基准版本尚未生成 OCR 快照；当前通过 `GET 200 + status=not_generated` 透出，不再表现为浏览器层 404 噪音
- `TEMPLATE_OCR_ANALYSIS_FAILED`
  - OCR 引擎、图像解码或运行时失败
- `TEMPLATE_PREVIEW_GENERATION_FAILED`
  - 预览图生成失败
- `MASK_REGION_OUT_OF_RANGE`
  - 草稿或正式 Mask 越界

### 7.2 排查顺序建议

1. 先确认 `Authorization` 和 `X-Workspace-Id`
2. 再确认 `template_id` 与 `baseline_revision_id` 是否匹配
3. 再确认视觉依赖是否安装完整
4. 最后用 `X-Request-Id` 配合后端日志排查
