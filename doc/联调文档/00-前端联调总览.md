# 前端联调总览

## 1. 适用范围

本文档面向当前 `VisionAutoTest` 前后端联调阶段，覆盖以下真实基线：

- 后端默认开发环境：`PostgreSQL + Alembic`
- 后端执行链路：`FastAPI + BackgroundTasks + Playwright`
- 当前已支持的编排与执行能力：
  - `wait`
  - `click`
  - `input`
  - `component_call`
  - `template_assert`
  - `ocr_assert`

本文档以当前后端代码实现、`doc/api/` 契约文档和本地联调环境为准。

## 2. 当前联调基线

- 接口前缀：`/api/v1`
- 健康检查：`GET /healthz`
- 默认管理员账号：`admin`
- 默认管理员密码：以本地 `backend/.env` 中 `VAT_DEFAULT_ADMIN_PASSWORD` 为准
- 工作空间内接口统一要求：
  - `Authorization: Bearer <token>`
  - `X-Workspace-Id`

当前后端已支持的真实执行步骤：

- `wait`
- `click`
- `input`
- `component_call`
- `template_assert`
- `ocr_assert`

其中：

- `component_call` 在执行时会展开为真实线性步骤
- `template_assert` 会产出基准图 / 实际图 / 差异图证据
- `ocr_assert` 会产出 OCR 截图证据

## 3. 本地环境约定

### 3.1 数据库

前后端本地联调默认使用 PostgreSQL：

- 管理库：`postgres`
- 开发库：`AutoTestDev`
- 默认账号：`postgres`
- 默认密码：`postgres`
- 端口：`5432`

本地业务库连接示例：

```text
postgresql+psycopg://postgres:postgres@127.0.0.1:5432/AutoTestDev
```

### 3.2 后端初始化

联调前请先完成：

```bash
cd backend
source .venv/bin/activate
python -m app.db.bootstrap
uvicorn app.main:app --reload
```

该初始化流程会：

1. 自动创建开发库（若不存在）
2. 执行 Alembic 迁移
3. 初始化默认管理员

### 3.3 视觉依赖

当前联调若涉及 `template_assert / ocr_assert`，后端运行机必须具备：

- `opencv-python-headless`
- `paddleocr`
- `paddlepaddle`
- `playwright` 浏览器依赖

建议安装方式：

```bash
cd backend
source .venv/bin/activate
pip install -e ".[dev]"
playwright install chromium
```

若缺少视觉依赖：

- 普通 `wait/click/input` 链路仍可运行
- `template_assert / ocr_assert` 会在执行期返回 `error`

## 4. 最近后端变更对前端的影响

### 4.1 编排能力扩展

后端已新增两类真实可执行步骤：

- `template_assert`
- `ocr_assert`

前端在步骤编辑、步骤回显、执行详情页需要按新增能力联调。

### 4.2 执行状态语义更明确

当前后端已明确区分：

- `failed`：断言不通过
- `error`：依赖缺失、载荷非法、运行时异常

前端执行详情与结果提示不应再把两者混为同一种失败。

### 4.3 证据链返回增强

`step-results` 现在可能返回以下媒体关联字段：

- `expected_media_object_id`
- `actual_media_object_id`
- `diff_media_object_id`

前端联调建议：

- `template_assert`
  - 展示基准图、实际图、差异图
- `ocr_assert`
  - 展示 OCR 实际截图
- 其余步骤
  - 至少展示 `actual_media_object_id` 对应截图（若后端写入）

## 5. 推荐联调顺序

### 5.1 普通执行链路

1. 登录
2. 创建工作空间
3. 创建环境档案
4. 创建组件与组件步骤
5. 创建测试用例与步骤
6. 创建测试套件并绑定用例
7. 将资源调整到可执行状态
8. 创建执行批次并轮询结果

### 5.2 视觉断言链路

若要联调 `template_assert / ocr_assert`，建议在普通链路基础上增加：

1. 上传媒体对象
2. 创建模板并生成当前基准版本
3. 在步骤中引用模板或 OCR 载荷
4. 执行后查看 `step-results` 中的证据媒体字段

## 6. 前端页面联调关注点

### 6.1 步骤编辑器

前端需要支持以下步骤载荷：

- `wait`
  - `payload_json.ms`
- `click`
  - `payload_json.selector`
- `input`
  - `payload_json.selector`
  - `payload_json.text`
- `component_call`
  - `component_id`
- `template_assert`
  - `template_id`
  - 可选 `payload_json.threshold`
- `ocr_assert`
  - `payload_json.selector`
  - `payload_json.expected_text`
  - 可选 `payload_json.match_mode`
  - 可选 `payload_json.case_sensitive`

### 6.2 执行入口提示

- 套件不是 `active` 时，禁用“立即执行”
- 用例不是 `published` 时，提示不可执行
- 被引用组件不是 `published` 时，提示不可执行
- `template_assert` 引用模板时，模板必须：
  - 属于当前工作空间
  - 存在当前基准版本
  - `match_strategy=template`
- `ocr_assert` 若关联模板，模板必须：
  - 属于当前工作空间
  - 存在当前基准版本
  - `match_strategy=ocr`

### 6.3 执行详情展示

- `case-runs`：展示用例级结果
- `step-results`：展示真实执行步骤
- 对于 `component_call`，前端先按线性步骤列表展示即可
- 对于视觉断言步骤：
  - `failed` 时高亮展示差异证据或 OCR 实际截图
  - `error` 时展示错误信息，不应误显示为业务断言失败

## 7. 当前仍不在联调范围内

- 组件内部再次嵌套 `component_call`
- JWT 正式鉴权
- Celery + Redis 异步执行集群
- MySQL 运行时兼容验证

## 8. 联调结论

当前提交给前端联调的后端口径为：

- PostgreSQL 开发联调已稳定
- 普通执行链路可联调
- `template_assert / ocr_assert` 后端已支持，可开始前端接入
- 执行详情页需同步适配 `failed / error` 语义与新证据链字段
