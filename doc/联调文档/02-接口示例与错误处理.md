## 1. 后端初始化示例

```bash
cd backend
source .venv/bin/activate
python -m app.db.bootstrap
```

- 自动执行 Alembic 迁移

## 2. 登录

### 2.1 请求

```http
POST /api/v1/sessions
Content-Type: application/json

{
  "username": "admin",
  "password": "admin123456"
}
```

### 2.2 响应

```json
{
  "data": {
    "session_id": "ses_xxx",
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.xxx.yyy",
    "refresh_token": "rfr_xxx",
    "token_type": "Bearer",
    "expires_in": 7200,
    "user": {
      "id": 1,
      "username": "admin",
      "display_name": "Administrator"
    }
  },
  "meta": {
    "request_id": "req_xxx"
  }
}
```

说明：

- 上述 `password` 仅为示例值，实际联调请以本地 `backend/.env` 中 `VAT_DEFAULT_ADMIN_PASSWORD` 为准
- `access_token` 已升级为 JWT，前端仍继续使用 `Authorization: Bearer <token>`

### 2.3 令牌续期

```http
POST /api/v1/session-refreshes
Content-Type: application/json

{
  "refresh_token": "rfr_xxx"
}
```

成功示例：

```json
{
  "data": {
    "access_token": "new_jwt_token",
    "refresh_token": "new_refresh_token",
    "token_type": "Bearer",
    "expires_in": 7200
  },
  "meta": {
    "request_id": "req_xxx"
  }
}
```

说明：

- 前端会在以下场景自动调用该接口：
  - 进入受保护路由前发现 access token 已过期或即将过期
  - 普通业务请求返回 `401` 且属于可恢复的登录态
  - access token 到期前的前置定时续期
- 正常续期成功后，前端会静默更新本地 `access_token / refresh_token / expires_in`，不弹成功提示。

### 2.4 当前会话校验

```http
GET /api/v1/sessions/current
Authorization: Bearer <token>
```

成功示例：

```json
{
  "data": {
    "session_id": "ses_xxx",
    "status": "active",
    "expires_at": "2026-03-30T14:30:00+00:00",
    "user": {
      "id": 1,
      "username": "admin",
      "display_name": "Administrator"
    }
  },
  "meta": {
    "request_id": "req_xxx"
  }
}
```

说明：

- 前端会在应用启动恢复历史会话时调用该接口，确认本地 session 与服务端锚点仍一致。
- 若接口返回 `TOKEN_REVOKED` 或 `SESSION_NOT_FOUND`，前端会提示后清理本地会话并跳回登录页。

## 3. 上传媒体并创建模板

### 3.1 上传媒体对象

```http
POST /api/v1/media-objects
Authorization: Bearer <token>
X-Workspace-Id: 1
Content-Type: multipart/form-data
```

表单字段：

- `file`: 图片文件
- `usage`: `baseline`
- `remark`: 可选

### 3.2 创建模板

```http
POST /api/v1/templates
Authorization: Bearer <token>
X-Workspace-Id: 1
Content-Type: application/json

{
  "template_code": "tpl_login_page",
  "template_name": "登录页模板",
  "template_type": "page",
  "match_strategy": "template",
  "threshold_value": 0.95,
  "status": "published",
  "original_media_object_id": 5001
}
```

说明：

- 创建模板时会自动生成首个基准版本
- 当前 `match_strategy` 仅支持 `template` 与 `ocr`
- 当前模板状态建议使用 `draft / published / archived`，其中仅 `published` 模板可用于执行
- `template_assert` 要求模板 `match_strategy=template`
- `ocr_assert` 若关联模板，则要求模板 `match_strategy=ocr`

### 3.3 采纳失败证据为新基准版本

```http
POST /api/v1/templates/6001/baseline-revisions
Authorization: Bearer <token>
X-Workspace-Id: 1
Content-Type: application/json

{
  "media_object_id": 7002,
  "source_type": "adopted_from_failure",
  "source_report_id": 8001,
  "source_case_run_id": 6002,
  "source_step_result_id": 9102,
  "remark": "accept current page as new baseline",
  "is_current": true
}
```

说明：

- `media_object_id` 必须等于失败步骤的 `actual_media_object_id`
- 当前不允许直接采纳 `diff_media_object_id`
- 当 `source_type=adopted_from_failure` 时，`source_step_result_id` 必填

## 5. 创建执行批次

```http
POST /api/v1/test-runs
Authorization: Bearer <token>
X-Workspace-Id: 1
Idempotency-Key: run-001
Content-Type: application/json

{
  "test_suite_id": 3001,
  "environment_profile_id": 4001,
  "trigger_source": "manual"
}
```

成功响应要点：

- 初始 `status` 为 `queued`
- 后续通过 `GET /api/v1/test-runs/{id}` 轮询
- 断言失败最终可能返回 `failed` 或 `partial_failed`
- 运行时异常最终可能返回 `error`

## 8. 常见错误与排查

### 8.1 后端无法启动

常见原因：

- PostgreSQL 未启动
- `AutoTestDev` 尚未创建
- Alembic 迁移未执行
- 缺少视觉依赖但误以为会影响全部接口

建议排查顺序：

1. 确认 PostgreSQL 可连接
2. 执行 `python -m app.db.bootstrap`
3. 确认后端虚拟环境已安装项目依赖
4. 再次启动 `uvicorn app.main:app --reload`

### 8.2 登录失败

若默认管理员登录失败，优先排查：

- 初始化命令是否执行成功
- 当前连接的是否为 `AutoTestDev`
- 开发库是否被重建但未重新种子管理员

### 8.5 续期失败

#### refresh token 已过期

```json
{
  "error": {
    "code": "REFRESH_TOKEN_EXPIRED",
    "message": "Refresh token is expired.",
    "details": []
  },
  "meta": {
    "request_id": "req_xxx"
  }
}
```

#### refresh token 已吊销

```json
{
  "error": {
    "code": "REFRESH_TOKEN_REVOKED",
    "message": "Refresh token is revoked.",
    "details": []
  },
  "meta": {
    "request_id": "req_xxx"
  }
}
```

补充约定：

- 登出成功后，同一会话下后续任意 refresh token 请求统一返回 `REFRESH_TOKEN_REVOKED`
- 该规则优先于“旧 refresh token 已消费”的 `REFRESH_TOKEN_INVALID` 语义，前端无需区分当前持有的是最新令牌还是会话内旧令牌

## 9. 联调口径说明

- 前端本地联调默认只关注开发库 `AutoTestDev`
- 测试数据库属于后端测试阶段独立使用，不纳入前端联调流程
- 当前后端已支持 `template_assert / ocr_assert`
- 当前后端不支持 `orb / mixed` 作为可交付模板策略
- 前端提交联调时，重点适配：
  - 登录响应新增 `token_type`
  - `session-refreshes` 的细分错误码 `REFRESH_TOKEN_INVALID / REFRESH_TOKEN_EXPIRED / REFRESH_TOKEN_REVOKED`
  - 模板策略与模板状态的受限枚举
  - `failed / error` 状态语义
  - 报告摘要结构化字段
  - `report-artifacts` 的 `artifact_type / case_run_id / step_result_id`
  - `artifact_url` 当前返回稳定访问路径，而不是对象键
