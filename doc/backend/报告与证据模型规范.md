# 报告与证据模型规范

## 1. 失败原因码（Failure Reason Code）

### 1.1 用例级失败原因码

| 错误码 | 场景 | 说明 |
|---|---|---|
| `ASSERTION_FAILED` | failed | 断言失败（默认） |
| `TEMPLATE_ASSERTION_FAILED` | failed | 模板匹配断言失败 |
| `OCR_ASSERTION_FAILED` | failed | OCR文本断言失败 |
| `STEP_EXECUTION_ERROR` | error | 步骤执行异常（默认） |
| `STEP_NOT_SUPPORTED` | error | 步骤类型不支持 |
| `STEP_CONFIGURATION_INVALID` | error | 步骤配置非法 |
| `STEP_EXECUTION_TIMEOUT` | error | 步骤执行超时 |
| `BROWSER_EXECUTION_ERROR` | error | 浏览器执行异常 |
| `SCREENSHOT_CAPTURE_FAILED` | error | 截图捕获失败 |
| `TEST_RUN_CANCELLED` | cancelled | 执行被取消 |
| `TEST_RUN_EXECUTION_ERROR` | error | 执行批次异常（默认） |

### 1.2 使用规则

- `case_run.failure_reason_code` 在 `failed/error/cancelled` 状态下必须非空
- `case_run.failure_summary` 在 `failed/error/cancelled` 状态下必须非空
- 前端不应从步骤明细反推失败原因，应直接消费 `failure_reason_code` 和 `failure_summary`

## 2. 报告产物分类（Artifact Type）

| 产物类型 | 说明 | 关联字段 |
|---|---|---|
| `run_screenshot` | 执行批次截图 | `case_run_id` |
| `step_actual` | 步骤实际截图 | `case_run_id`, `step_result_id` |
| `step_diff` | 步骤差异图 | `case_run_id`, `step_result_id` |
| `step_ocr` | OCR识别截图 | `case_run_id`, `step_result_id` |

### 2.1 使用规则

- `report_artifacts.artifact_type` 仅允许上述4种类型
- `artifact_url` 统一返回 `/api/v1/media-objects/{media_object_id}/content`
- 前端不应依赖文件名或备注推断证据来源，应使用 `case_run_id` 和 `step_result_id` 定位

## 3. 报告摘要结构（Summary JSON）

### 3.1 标准结构

```json
{
  "status": "failed",
  "counts": {
    "total": 10,
    "passed": 7,
    "failed": 2,
    "error": 1,
    "cancelled": 0
  },
  "failure": {
    "code": "TEMPLATE_ASSERTION_FAILED",
    "summary": "Template matching failed with score 0.7234 below threshold 0.85"
  },
  "timing": {
    "started_at": "2026-03-30T05:00:00Z",
    "finished_at": "2026-03-30T05:12:34Z",
    "duration_ms": 754000
  },
  "artifacts": {
    "total": 45,
    "by_type": {
      "run_screenshot": 10,
      "step_actual": 20,
      "step_diff": 10,
      "step_ocr": 5
    }
  }
}
```

### 3.2 字段规则

| 字段 | 必填 | 说明 |
|---|---|---|
| `status` | 是 | 必须与 `test_run.status` 一致 |
| `counts` | 是 | 用例统计，所有字段必填 |
| `failure` | 条件 | `passed` 时为 `null`，其他状态必须非空 |
| `failure.code` | 条件 | 失败时必填，使用标准失败原因码 |
| `failure.summary` | 条件 | 失败时必填，提供可读的失败描述 |
| `timing` | 是 | 时间统计，所有字段必填 |
| `artifacts` | 是 | 产物统计，所有字段必填 |
| `artifacts.by_type` | 是 | 按类型统计，仅包含标准产物类型 |

### 3.3 状态与 failure 字段对应关系

| status | failure 字段 |
|---|---|
| `passed` | `null` |
| `failed` | 必须非空 |
| `partial_failed` | 必须非空 |
| `error` | 必须非空 |
| `cancelled` | 必须非空 |

## 4. 前端消费指南

### 4.1 失败原因展示

```typescript
// 直接使用 case_run 的失败信息
if (caseRun.status === 'failed' || caseRun.status === 'error') {
  displayFailure({
    code: caseRun.failure_reason_code,
    summary: caseRun.failure_summary
  })
}
```

### 4.2 报告摘要展示

```typescript
// 从 summary_json 获取结构化信息
const summary = report.summary_json
displayStatus(summary.status)
displayCounts(summary.counts)
if (summary.failure) {
  displayFailure(summary.failure)
}
displayTiming(summary.timing)
displayArtifacts(summary.artifacts)
```

### 4.3 证据链访问

```typescript
// 使用 artifact_url 直接访问
artifacts.forEach(artifact => {
  const url = artifact.artifact_url
  // url 格式: /api/v1/media-objects/{id}/content
  displayImage(url)
})
```
