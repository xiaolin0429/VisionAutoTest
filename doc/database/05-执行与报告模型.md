# 执行与报告模型

## 1. 表清单

| 表名 | 说明 |
|---|---|
| `exec_test_runs` | 执行批次表 |
| `exec_test_case_runs` | 用例执行实例表 |
| `exec_step_results` | 步骤结果表 |
| `report_run_reports` | 报告主表 |
| `report_artifacts` | 报告证据链表 |

## 2. 表设计

### 2.1 exec_test_runs

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | BIGINT | PK | 主键 |
| `workspace_id` | BIGINT | NOT NULL, FK | 工作空间 ID |
| `test_suite_id` | BIGINT | NOT NULL, FK | 套件 ID |
| `environment_profile_id` | BIGINT | NOT NULL, FK | 环境档案 ID |
| `device_profile_id` | BIGINT | NULL, FK | 设备预设 ID |
| `trigger_source` | VARCHAR(32) | NOT NULL | `manual/ci/schedule/webhook` |
| `triggered_by` | BIGINT | NULL, FK | 触发人 |
| `idempotency_key` | VARCHAR(128) | NULL | 幂等键 |
| `status` | VARCHAR(32) | NOT NULL | 执行状态 |
| `started_at` | TIMESTAMP | NULL | 开始时间 |
| `finished_at` | TIMESTAMP | NULL | 结束时间 |
| `total_case_count` | INTEGER | NOT NULL DEFAULT 0 | 总用例数 |
| `passed_case_count` | INTEGER | NOT NULL DEFAULT 0 | 通过数 |
| `failed_case_count` | INTEGER | NOT NULL DEFAULT 0 | 失败数 |
| `error_case_count` | INTEGER | NOT NULL DEFAULT 0 | 异常数 |
| `created_at` | TIMESTAMP | NOT NULL | 创建时间 |
| `updated_at` | TIMESTAMP | NOT NULL | 更新时间 |

索引建议：

- `uk_exec_test_runs_idempotency(workspace_id, idempotency_key)`
- `idx_exec_test_runs_status_created(status, created_at)`

### 2.2 exec_test_case_runs

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | BIGINT | PK | 主键 |
| `test_run_id` | BIGINT | NOT NULL, FK | 执行批次 ID |

| `sort_order` | INTEGER | NOT NULL | 执行顺序 |

### 2.3 exec_step_results

| 字段 | 类型 | 约束 | 说明 |

| 字段 | 类型 | 约束 | 说明 |

## 3. 设计说明

- 执行与报告为典型流水域，不做逻辑删除。
- `exec_step_results` 存储真实执行后的线性步骤序列；若用例步骤中引用 `component_call`，组件步骤会在运行时展开后再落库。
- 创建执行批次前，应用层需先校验：套件为 `active`，套件内测试用例与引用组件为 `published`。
- 报告主表只存聚合信息，大文件通过 `report_artifacts` 关联。
- 评分、耗时、失败码等字段拆开存储，便于统计分析。
- `report_run_reports.summary_json` 当前按结构化摘要存储，至少包含 `status`、`counts`、`failure`、`timing`、`artifacts` 五组信息。
- 当前 `report_artifacts.artifact_type` 收口为 `run_screenshot/step_actual/step_diff/step_ocr`。
- `report_artifacts` 同时承担报告级证据索引职责，可通过 `case_run_id` 与 `step_result_id` 反向定位执行上下文。
