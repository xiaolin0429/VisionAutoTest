# 数据库设计总览

## 1. 设计目标

数据库设计必须满足以下要求：

- 满足多租户隔离和审计追踪要求。
- 兼容工作空间级配置、模板资产、测试编排和执行流水。
- 满足行业通用表设计规范，便于迁移脚本、索引治理和后续容量扩展。

## 2. 命名规范

### 2.1 表命名

- 统一使用 `snake_case`
- 统一使用“领域前缀 + 复数资源名”
- 表名示例：`iam_users`、`core_workspaces`、`exec_test_runs`

### 2.2 字段命名

- 主键统一：`id`
- 外键统一：`{resource}_id`
- 状态字段统一：`status`
- 排序字段统一：`sort_order`
- 版本字段统一：`version_no`
- 逻辑删除字段统一：`is_deleted`

## 3. 通用字段规范

### 3.1 业务主表通用字段

以下字段规范默认适用于“可编辑、可审计的主业务表”。纯关系表、步骤明细表、系统生成快照和执行流水表可按场景做精简。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | BIGINT | PK | 代理主键 |
| `created_at` | TIMESTAMP | NOT NULL | 创建时间，UTC |
| `created_by` | BIGINT | NULL | 创建人 |
| `updated_at` | TIMESTAMP | NOT NULL | 更新时间，UTC |
| `updated_by` | BIGINT | NULL | 更新人 |
| `version_no` | INTEGER | NOT NULL DEFAULT 1 | 乐观锁版本号 |
| `is_deleted` | BOOLEAN | NOT NULL DEFAULT FALSE | 逻辑删除标识 |

### 3.2 执行流水表规范

执行流水类表不使用逻辑删除，保留真实历史，避免审计断链。

## 4. 类型规范

- 标识符优先使用 `BIGINT`
- 编码类字段使用 `VARCHAR(64)`
- 名称类字段使用 `VARCHAR(128)`
- 描述类字段使用 `VARCHAR(512)` 或 `TEXT`
- 金额、计数、阈值使用显式数值类型，不使用字符串承载
- 半结构化内容仅在必要时使用 `JSON`

## 5. 索引规范

- 每个外键字段必须评估并建立索引。
- 唯一业务编码必须建立唯一索引。
- 高选择性查询字段组合建立联合索引。
- 长文本字段不参与普通索引。
- 对逻辑删除主表，唯一索引建议采用业务字段 + `is_deleted` 组合。

## 6. 约束规范

- 主表必须有主键。
- 关系表必须有唯一约束防止重复绑定。
- 枚举字段应在应用层和数据库层双重约束。
- 外键引用必须明确 `ON UPDATE` / `ON DELETE` 策略。

## 7. 审计与安全规范

- 敏感配置如密钥、Token、密码不得明文存储。
- 执行日志、Webhook 投递、通知记录需可审计。
- 重要状态变更建议配套操作日志表，后续可扩展。
